### Задание 1. СУБД

Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.    
Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?    

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.    
1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?        
1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.    
1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?    
1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.    
1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?    
1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.    
1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?    
1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?    
Приведите ответ в свободной форме.  

## Решение 1
1.1. Для бюджетирования проектов и финансовых отчетов лучше всего подойдет реляционная СУБД, такая как PostgreSQL или Oracle. Они обеспечивают целостность данных, поддержку сложных запросов и транзакций.    
1.1.* Для ускорения хеширования можно использовать API, такие как Redis или Memcached, которые предоставляют быструю кэш-память для ускорения обработки данных.    
1.2. Для лендингов и CRM лучше всего подойдет NoSQL СУБД, например, MongoDB, так как она обеспечивает гибкость в структуре данных и высокую скорость обработки.    
1.2.* Эту задачу можно закрыть одной СУБД, например, с использованием Firebase, которая поддерживает и лендинги, и CRM, благодаря встроенным функциям для работы с пользователями и данными в реальном времени.    
1.3. Для базы норм и правил можно использовать ту же СУБД, что и в предыдущих задачах. Например, MongoDB позволит легко структурировать и изменять данные, сохранив простоту доступа.    
1.3.* Реализация может быть выполнена через отдельные коллекции в одной и той же базе данных, что облегчит управление.    
1.4. Для задач логистики будет подходить реляционная СУБД, такая как PostgreSQL, поскольку она хорошо работает с взаимосвязями и позволяет проводить сложные запросы для распределения курьеров.    
1.4.* Отдел закупок может быть подключен к той же СУБД, если их данные взаимосвязаны, иначе лучше создать отдельную, интегрируемую с логистикой, например, на базе MySQL.    
1.5.* Да, все задачи можно решить с помощью одной СУБД. Например, PostgreSQL гибко справится со всеми перечисленными задачами и обеспечит целостность данных.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.    
2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?    
Приведите ответ в свободной форме.         

## Решение 2
2.1. Для успешного пополнения баланса телефона необходимо выполнить следующие шаги:    
Идентификация пользователя: Подтверждение его учетных данных.     
Выбор метода платежа: Определение способа, которым будет выполнено пополнение.    
Подсчет суммы: Проверка доступных средств пользователя.    
Выполнение транзакции: Начало процесса перевода средств.    
Подтверждение транзакции: Заявление о завершении транзакции от процессингового центра.    
Обновление баланса: Применение новой суммы к счету пользователя.    
2.1.* При автоплатеже действия немного изменятся:    
Проведение мониторинга баланса,   
Автоматический запуск пополнения без подтверждения пользователя,    
Уведомление о выполненной операции через приложение или SMS.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.    
3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.    
Приведите ответ в свободной форме.    

## Решение 3

3.1. Пять преимуществ SQL-систем по сравнению с NoSQL:     
Строгая схема данных: Обеспечивает целостность и легко управляет изменениями.    
Транзакционная поддержка: Полная поддержка ACID-транзакций.    
Сложные запросы: Поддерживают сложные SQL-запросы и соединения таблиц.    
Устойчивость к данным: Защита данных от утечек и ошибок.    
Широкое распространение: Большие сообщества пользователей и разработчиков, что упрощает поиск решения. 

3.1.* Преимущества NewSQL систем:    
Скорость обработки: Высокая производительность при сохранении поддержки ACID.    
Горизонтальная масштабируемость: Возможность распределения нагрузки на множество серверов.    
Совместимость с SQL: Возможность использования привычных SQL-запросов.     
Современные архитектуры: Оптимизированы для облачных и распределенных сред.    
Гибкость обработки данных: Поддерживают разные парадигмы хранения данных.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.    
На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?    
Приведите ответ в свободной форме.    

## Решение 4
Для выбора типа СУБД при работе с огромными объемами данных и выделением 1000 машин следует учитывать следующие критерии:    

Тип данных: Если данные структурированы, лучше использовать реляционные СУБД (например, PostgreSQL). Если данные полуструктурированы или неструктурированы, предпочтительнее NoSQL решения (например, Apache Cassandra или MongoDB).     
Масштабируемость: СУБД должна поддерживать горизонтальную масштабируемость для обработки большого объема запросов и данных. Это позволит эффективно использовать 1000 машин.    
Производительность: Выбор СУБД зависит от требований к быстродействию. Важно, чтобы она обеспечивала высокую скорость операций чтения и записи.     
Поддержка параллельных вычислений: СУБД должна эффективно работать с параллельными запросами, чтобы максимально использовать ресурсы кластера.     
Обработка больших данных: Необходима интеграция с инструментами, которые обеспечивают обработку больших данных (например, Apache Hadoop).     
Лучшая модель распределённых вычислений для этой задачи — Apache Spark. Она поддерживает обработку данных в режиме реального времени, возможность выполнения параллельных вычислений и масштабируемость, что делает её идеальной для работы с огромными объемами данных на 1000 машинах.
